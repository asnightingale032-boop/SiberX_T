# Reflection: How AI-Assisted Development Impacted This Project's Security

**Assessed deployment:** https://siberx-t.vercel.app/

This document is written from the position of the tool that built the thing being assessed, which makes it an unusually direct case study rather than a general commentary. Three things are worth being honest about: what AI assistance made *better*, what it made *worse* or missed, and what that implies about how AI-assisted projects should actually be reviewed.

## What sped up, and what that speed cost

The site went through three real iterations: an initial build, an explicit "harden this" pass, and now this assessment. That sequence is itself the finding. **The first version was functional and reasonably careful, but it was not the version you'd want in production.** It loaded fonts from Google's CDN (a third-party network dependency with no security upside), used an inline `<script>` block on the login page, and had at least one interactive control wired up with an inline `onclick="..."` attribute. None of that is *dramatically* dangerous on its own, but together it's exactly the kind of accumulated looseness that makes a strict Content-Security-Policy impossible to adopt later without a rewrite — and a CSP that permits `unsafe-inline` scripts is one of the more common real-world misconfigurations precisely because retrofitting it is annoying enough that teams don't bother.

That looseness didn't happen because the model "didn't know better" — it happened because *security wasn't the stated objective of the first pass*. The first request was "build me a crisis-simulation website." The second was explicitly "make sure this passes a vulnerability assessment." AI-assisted development is extremely responsive to what's asked for, which is a double-edged property: once security was named as a requirement, dozens of files got escaping helpers, a strict CSP, sanitization, and session expiry applied *consistently* across the whole codebase in one pass — a level of uniform coverage that's genuinely hard for a human to match by hand across nine files without missing one. But that same responsiveness means an unstated requirement stays unmet. If the assignment brief hadn't explicitly asked for a security pass, the looser first version is what would have shipped, and it would have looked equally polished and complete to someone not specifically looking for these gaps.

**The practical implication: security has to be stated as an explicit requirement, every time, not assumed as a default.** "Build X" and "build X securely" produce measurably different code from the same tool, and treating the first as if it implies the second is the mistake this exercise is designed to surface.

## What a second pass caught that the first hardening pass didn't

This assessment (the third pass) found one genuine, previously-unnoticed bug: `logout()` cleared the signed-in user but left every other piece of exercise state — case notes, chat transcripts, decision history, lab scores — sitting in `localStorage`, so a second person on the same shared browser would inherit the first person's data. This survived an entire dedicated hardening pass that added escaping, sanitization, CSP, MFA, and session expiry to the same file that contained the bug.

That's not a coincidence, and it's worth naming why. The hardening pass was organized around a checklist of *categories* — exposed secrets, input validation, XSS, auth, misconfig — and it was thorough within each category. But the logout bug isn't really an instance of any single category; it's a **cross-cutting state-lifecycle issue**: does the full *set* of fields that got added to the shared state object over several files and several revisions all get cleared at the one point where a session ends? Answering that requires holding the whole data model in view at once and asking a structural question, not scanning line-by-line for a known bad pattern. That's a harder failure mode to catch, for a model or a person, because nothing about any individual line looks wrong — `s.user = null` is completely reasonable-looking code. The bug is only visible from the outside, by asking "what does logout actually promise, and does the implementation deliver on that promise in full," which is a different kind of review than pattern-matching against a checklist.

This matches a broader, well-documented pattern with AI-generated code: **it's strong at applying a known-correct pattern consistently once that pattern is specified** (escape-on-render, sanitize-on-entry, no inline handlers — all applied uniformly, all verified correct in this review), **and comparatively weaker at noticing when a feature's implicit contract has silently drifted** as the surrounding code grew. A checklist-driven review, whether done by AI or a human working from a checklist, will systematically miss this category of issue, because checklists are enumerations of known failure shapes, and "the code technically does what it says but the thing it says is incomplete" isn't a shape, it's a gap in the promise itself.

## What this implies for how AI-assisted projects should actually be reviewed

Three concrete takeaways, matching what this assignment is built to demonstrate:

1. **State security requirements explicitly and early**, not as an afterthought — the difference in output quality between "build X" and "build X securely" is large and consistent, and waiting to ask until after the fact means shipping the looser version at least once.
2. **A checklist-based security pass is necessary but not sufficient.** It reliably catches known failure shapes (secrets, injection, missing headers) and should absolutely be run — but it should be paired with at least one review pass that asks structural questions about state, lifecycle, and promises made by function names and UI copy, not just pattern-matching against a list.
3. **Treat "it looks thorough" as a weak signal.** The hardened version of this app looked, and largely was, meaningfully more secure than the first draft — consistent escaping, a real CSP, session expiry, sanitized inputs. That genuine improvement is exactly what makes it easy to stop looking. The bug that slipped through did so *despite* real rigor elsewhere, not because the rigor was fake. Confidence from a clean-looking pass is not the same thing as confidence from having asked the right question.

None of this is an argument against AI-assisted development — the speed and consistency gains here are real and were demonstrated across dozens of files in a single request. It's an argument for treating "AI helped build it fast" and "a dedicated adversarial review happened afterward" as two separate, both-necessary steps, exactly as this assignment models.
